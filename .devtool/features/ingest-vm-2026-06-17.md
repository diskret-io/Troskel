# Ingest VM: full architectural fix for host-side file-USB mount

The file USB's filesystem is currently parsed by the **host** kernel before the Firecracker boundary is crossed. This exposes the host's filesystem-driver subsystem to attacker-chosen input. Phase 0 of the roadmap (the work already shipped at 0.9.0) narrowed this surface substantially: ext4 / vfat only, explicit `-t`, `ro,nodev,nosuid,noexec`, private mount namespace, nested-mount refusal, staging-walk symlink-race defences. But the host kernel still parses the filesystem. NTFS and exFAT remain unsupported because their parsers carry too much surface.

The ingest-VM work moves filesystem parsing into a dedicated Firecracker microVM. The host never mounts the file USB. The file USB attaches to a small ingest VM whose kernel parses the filesystem; the ingest VM extracts files into a clean ext4 image that it presents to the scanner VM(s) on a separate virtio device. The host coordinates but does not parse.

This is the major architectural improvement scheduled for 1.2.0.

Roadmap doc: `docs/roadmap/ingest-vm.md`.

## Why it matters

After this lands:

1. NTFS, exFAT, and any other filesystem the ingest VM's kernel supports become safely usable. The host-side allowlist constraint disappears.
2. A kernel-parsing exploit in the ingest VM is contained by the Firecracker boundary, exactly like an engine-parsing exploit in the scanner VM.
3. The Tier 2 remediation plan's headline architectural claim (no host kernel ever sees hostile filesystem input) becomes true rather than approximately-true.
