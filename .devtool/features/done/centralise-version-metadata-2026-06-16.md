# Centralise version metadata

The project version currently lives in seven string locations across
three files (README, the generator heredoc, and the committed
SBOM.json). The 0.9.0 to 1.0.0 bump audit found a stale `0.1.0`
reference that had survived multiple version bumps undetected: direct
evidence that hand-editing N copies does not converge, because nothing
forces the copies to agree.

## Motivation

Seven hand-maintained copies of a single fact means every bump is an
N-way edit with no check that the N agree. The failure mode is not
hypothetical: the audit already caught one copy two minor versions
stale. The risk a stale version carries here is not cosmetic. A
build-manifest or SBOM that reports the wrong version misattributes
which code and which signature set produced a given scan, which is a
provenance defect in a security-critical artefact.

## Target state

One canonical source for the version string. Every other location
derives from it at build time rather than carrying its own copy. After
the change, a version bump is a single edit, and any consumer that
needs the version reads or is generated from the canonical source.

## Acceptance criteria

- A single canonical version source exists (e.g. `config/versions.env`
  or an equivalent already in the tree; confirm the right home rather
  than introducing a new file if an existing one fits).
- The previously-enumerated seven locations no longer carry an
  independent literal. Each is either generated from, or reads from,
  the canonical source.
- A check that fails loudly on drift. Grepping the tree for a version
  literal that does not match the canonical source returns nothing, and
  this grep is wired into `make validate` (or the existing lint stage)
  so a reintroduced literal fails CI rather than surviving to a tag.
  This is the substantive guard: without it the change reduces the
  count from seven to one but does nothing to stop the count climbing
  back, which is the actual defect.
- The stale `0.1.0` survivor is confirmed eliminated, not merely
  outvoted: grep the tree for `0.1.0` and confirm zero hits outside
  legitimate changelog history.

## Failure mode to test

The regression test is the drift check itself: introduce a deliberate
mismatched version literal in a test fixture (or a throwaway branch)
and confirm `make validate` fails. A drift check that passes when a
mismatch is present is decorative. The test that proves the fix is the
one that would have caught the original `0.1.0` survivor at the commit
that introduced it.

## Implementation shapes

The superseded roadmap doc recorded two candidate implementation
shapes. That doc has been deleted and its content is not recoverable
from project state, so the two shapes are not reconstructed here rather
than guessed at. Re-derive the options at pickup from the actual tree:
the choice is broadly between build-time generation (canonical source,
all copies regenerated by the build) and read-at-runtime (consumers
read the canonical source directly). Decide based on which locations
are build artefacts versus committed files.

## Sequencing

Standalone; no dependencies on other cards. Cheap enough to land
opportunistically. Natural companion to the v1.0.0 release-gate card,
whose drift pass currently relies on a manual `0.1.0`/`0.9.1` grep as a
stopgap; this card is the durable replacement for that stopgap. Target:
land in the same commit as whatever first triggers the next version
bump after 0.9.1, plausibly the 1.0.0 tag itself.
