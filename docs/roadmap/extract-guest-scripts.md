# Extract embedded guest scripts from `build-scanner-image.sh`

`scripts/build-scanner-image.sh` currently contains the guest's `run-scan.sh` (the in-VM scan entrypoint) and `inittab` as heredocs. `run-scan.sh` is now around 100 lines of shell with engine-specific logic for ClamAV and LOKI-RS, plus verdict combination, non-trivial code wrapped inside a `cat > ... <<'SCAN'` block.

Same problem and same solution as the Butane extraction (see `extract-butane-scripts.md`): heredoc-embedded code is harder to edit, harder to lint, harder to read. The guest scripts deserve to live as their own files.

## What changes

Move the heredoc bodies to `guest/` at the project root:

```
guest/
  run-scan.sh              ← the in-VM scan entrypoint
  inittab                  ← busybox init configuration
```

`build-scanner-image.sh` replaces the heredocs with `cp` operations:

```bash
cp "${SCRIPT_DIR}/../guest/run-scan.sh" "${WORK}/usr/local/bin/run-scan.sh"
chmod +x "${WORK}/usr/local/bin/run-scan.sh"
cp "${SCRIPT_DIR}/../guest/inittab" "${WORK}/etc/inittab"
```

The actual content stays identical. Only the source location changes.

## Side effects

- `guest/run-scan.sh` becomes shellcheckable. The current `set -eu` and busybox-portable `/bin/sh` constraints continue to apply, so the file's first line stays `#!/bin/sh`.
- The capa roadmap task (`capa-third-engine.md`) becomes meaningfully easier: the per-file invocation loop and the JSON parser for capa's output go into `guest/run-scan.sh` directly instead of into another heredoc inside another already-long script.
- A directory listing of the project finally reflects the architectural three-way split: `scripts/` runs on the build station, `config/host-scripts/` runs on the scanning host (after the Butane extraction lands), `guest/` runs inside the Firecracker microVM. Useful for orientation.

## What stays the same

The resulting `scanner-rootfs.ext4` is byte-identical to before. Behaviour does not change.

## Estimated effort

Two to three hours. Smaller than the Butane extraction because there are only two files and `cp` is simpler than `local:`. Verify by building a rootfs both ways and checking the SHA-256 of `scanner-rootfs.ext4` matches.

## Sequencing

Lower-priority than the Butane extraction. Both are independent, neither blocks the other — but the Butane extraction has higher payoff (more scripts moved, removes the test-scan.sh extraction logic). Do this one second.

## Open questions

- Should `guest/` contain anything beyond these two files? The project currently has no other guest-side artefacts that aren't either downloaded (LOKI-RS binary, YARA rules, ClamAV signatures) or generated by the build process. If capa lands, its capability-to-verdict policy file is a candidate, but that decision belongs to the capa task, not this one.
