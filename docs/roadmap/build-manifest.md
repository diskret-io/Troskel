# Per-build manifest

Every scan that produces a verdict — green, red, or yellow — was produced by a specific build of the scanner against a specific set of detection inputs. Today, that "specific" is captured only loosely: ClamAV signatures have a date file, YARA rules have a date file, and the rest is inferred from whatever `versions.env` happened to say at build time. The build-manifest task formalises this into a single, structured, per-build record that an admin can use to reproduce a historical scan or explain a verdict that changed between two runs.

## Motivation

A scanner that runs on Tuesday and produces a green verdict, then runs on Wednesday and produces a red verdict on the same file, is not behaving incorrectly. Something changed between the two runs: a new ClamAV signature published overnight, a YARA Forge release that added a new rule, an IOC update. The change is the system working as designed — fresh detection inputs are the whole point.

What the system *doesn't* do today is tell the admin precisely what changed. The two date files (`signature-date`, `yara-rules-date`) say roughly when each detection input was refreshed, but they don't say which version was used or what its contents hashed to. So the admin cannot say with certainty "yesterday's scan used signatures X.Y.Z and today's used X.Y.W; rule N is in W but not Z, and rule N is what flagged this file". The signal exists in the world but the project doesn't capture it.

The build manifest captures it. For every build the scanner-image is produced from, a JSON document records:

- The exact software versions used (Firecracker, Butane, LOKI-RS, the kernel, the wordlist, CoreOS stream resolution).
- The exact detection input versions used (ClamAV `.cvd` version numbers, the resolved YARA Forge release tag, the signature-base IOC tag).
- The SHA-256 of every artefact, recorded at download time.
- How each artefact was verified (sidecar, GPG-bootstrap, TLS-only, etc.) — see the verification taxonomy below.
- The Troskel project commit that produced the build, including whether the tree was clean.
- Timestamps for the build and for each detection-input download.

The manifest travels with the data USB. After load, it lives at a stable location on the scanning host. An admin investigating a verdict can read it directly with `cat`; can compare two manifests with `diff`; can reconstruct an exact historical build by checking out the same Troskel commit and feeding the same `versions.env` into `run-update.sh`.

## What the manifest is not

The manifest is a per-build receipt, not a per-scan log. A given scanner-image is rebuilt at most once per signature refresh; one manifest covers every scan run against that image until the next refresh. Per-scan information — which file USB was scanned, when, with what verdict — is a different artefact and belongs in the output-USB clean certificate work (`output-usb.md`).

The manifest is not the SBOM. The SBOM (`sbom-automation.md`) is the project's inventory of *what can be in a build* — components, suppliers, licences. The manifest is the operational record of *what was in this build*. They overlap (both read `versions.env`, both include hashes), and they are generated by the same tooling for that reason, but they answer different questions and are read by different audiences.

### Field rationale

**`manifest_version`**: explicit schema version. The first emission is `"1"`. Bumped when the schema changes in a way that would confuse existing tooling. Forward compatibility through a known field is cheaper than guessing.

**`generated_at`**: ISO-8601 UTC, same format used throughout the project (`date -u --iso-8601=seconds`). When the manifest itself was emitted; this is roughly the build completion time.

**`build_environment.troskel_commit`**: the git SHA of the Troskel repository at the time of the build. Captured by `git rev-parse HEAD`. This is the single most useful field for reproduction — it pins the entire project's code state, including `versions.env`, the download scripts, and the build pipeline.

**`build_environment.troskel_dirty`**: boolean. True if `git status --porcelain` returned anything non-empty at build time, i.e. there were uncommitted changes. A build from a dirty tree is by definition unreproducible (the commit SHA doesn't fully describe the code that ran), and the manifest should record that fact rather than hide it. A dirty build can still be valid for operational use; it just shouldn't be claimed as reproducible.

**`build_environment.build_host_kernel`**: captured by `uname -sr`. Mostly informational. The build process itself isn't sensitive to the host kernel because debootstrap and the rest run in containers, but a record of "this is the machine where the build happened" is useful for diagnosing weird environmental failures after the fact.

**`build_environment.debian_release`**: from `DEBIAN_RELEASE` in `versions.env`. Pinned, but captured for completeness.

**`software.<component>.version`**: from `versions.env`, the pinned value.

**`software.<component>.sha256`**: from `versions.env` for components with recorded constants (Firecracker, Butane, LOKI-RS, wordlist); from the build state for the kernel (which uses record-at-first-download and writes its hash back to `versions.env`). The value here is the same as what's verified at download time; including it in the manifest means the manifest is self-contained for an audit conversation that may not have the contemporaneous `versions.env` to hand.

**`software.<component>.verification`**: how trust in this artefact was established. The taxonomy is:

| Value                      | Meaning                                                                                                                                                                                                                                                   |
|----------------------------|-----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|
| `sidecar`                  | Upstream publishes a `.sha256` sidecar per asset; verified at download against the sidecar from the same release URL.                                                                                                                                     |
| `gpg-bootstrap`            | Upstream publishes a GPG signature but no SHA-256 sidecar. SHA-256 was recorded once after a manual GPG verification; subsequent builds verify against the recorded value.                                                                                |
| `recorded`                 | A pinned SHA-256 was recorded by a developer at the time of pinning. Used for upstreams that don't publish per-asset integrity files (e.g. the EFF wordlist).                                                                                             |
| `record-at-first-download` | No upstream integrity file; the first build to run computed the hash and wrote it to `versions.env`; subsequent builds verify against the recorded value. Used for the guest kernel.                                                                      |
| `upstream-signed`          | The artefact carries an upstream signature verified by tooling outside Troskel (e.g. the CoreOS ISO verified by `coreos-installer` against the Fedora signing key). No SHA-256 is recorded by Troskel because the upstream tooling does the verification. |
| `tls-only`                 | No upstream integrity guarantee; trust rests on TLS-from-source-domain at download time. Used for YARA Forge, which publishes no sidecars or signatures. The SHA-256 is recorded for reproducibility but did not gate the download.                       |
| `freshclam-embedded-keys`  | ClamAV signatures, verified by `freshclam` against keys embedded in the ClamAV binary at download time. The `.cvd` files are individually signed; Troskel relies on the upstream tooling.                                                                 |
| `tag-pinning`              | The upstream is pinned to an immutable release tag (not a moving branch). Reproducibility rests on the tag's immutability; per-file hashes are not recorded. Used for the LOKI IOC base.                                                                  |

**`software.coreos_stream.resolved_version`**: the actual CoreOS release that the `stable` stream resolved to at build time. Captured by `coreos-installer` during ISO extraction; this is what makes the CoreOS dependency reproducible even though the stream is floating.

**`detection_inputs.clamav_signatures.files[].version`**: extracted from the `.cvd` header using `sigtool --info`. Each ClamAV signature file carries a monotonically increasing version number; capturing it lets an admin say precisely "this build used main.cvd version 62".

**`detection_inputs.yara_forge.resolved_tag`**: the GitHub release tag that the `latest` redirect resolved to at build time. Captured via a HEAD request to `https://github.com/YARAHQ/yara-forge/releases/latest` and parsing the `Location` header. Without this, "reproduce yesterday's scan" requires the admin to guess which YARA Forge release was current yesterday — solvable, but solvable badly. With it, the reproduction is mechanical.

**`detection_inputs.yara_forge.archive_sha256`**: hash of the downloaded archive, recorded at download time. There is no upstream value to verify this against (YARA Forge publishes nothing); the hash exists as a reproducibility receipt, not a verification gate.

**`detection_inputs.loki_ioc_base.files[].sha256`**: hashes of the three IOC text files. Tag-pinning already provides reproducibility (the tag is immutable), but per-file hashes are cheap to compute and provide an additional integrity record. Useful if the admin wants to verify that the files copied to the data USB match the files originally downloaded.

**`downloaded_at` timestamps**: per detection input. Distinct from `generated_at` because the manifest is emitted after the build is complete, but the detection inputs were downloaded earlier in the pipeline (potentially much earlier in a long build). Per-input timestamps let an admin see, for example, "ClamAV signatures were 14 minutes old when this build emitted; YARA rules were 13 minutes old". Useful for diagnosing freshness-related verdict differences.

## Generation

A single script — `scripts/generate-build-records.sh` — produces both the manifest and the SBOM. Combining them into one generator is deliberate: they read the same state, and the act of generating them separately introduces a coordination problem (did both run? did they see the same state?) that combining them removes.

The generator runs as the final step of `run-update.sh`, after every download and build step has completed. By that point, every value the manifest records is either in `versions.env`, in `/var/lib/troskel/`, or computable from a small number of well-defined sources (`git rev-parse`, `uname`, `curl` against `releases/latest`, `sigtool --info`).

The generator's responsibilities:

1. Source `config/versions.env` for pinned values and recorded hashes.
2. Read `/var/lib/troskel/signature-date`, `/var/lib/troskel/yara-rules-date`, and the YARA Forge archive hash from build-time logs.
3. Compute the CoreOS resolved version via `coreos-installer iso extract pxe` output.
4. Resolve the YARA Forge `latest` tag via a HEAD request.
5. Run `sigtool --info` against each ClamAV `.cvd` to extract version and build hash.
6. Run `git rev-parse HEAD` and `git status --porcelain` against the Troskel repo root.
7. Emit two files: `SBOM.json` (the existing CycloneDX inventory) and `/var/lib/troskel/build-manifest.json` (the new per-build record).

The two outputs are produced from the same in-memory state, so they cannot disagree on any value they both record.

### Side effects on download scripts

A few download scripts need small changes to capture state the generator will read:

- `download-loki-yara-rules.sh` already logs the YARA Forge archive SHA-256 to stdout. The generator needs a more durable home for it — write it to `/var/lib/troskel/yara-forge-archive-sha256` at download time.
- `download-clamav-signatures.sh` already runs after `freshclam`; the `.cvd` files are on disk by the time the generator runs, so no script change is needed — the generator just reads them.
- `download-kernel.sh` (modified in phase B of checksum-verification) already writes `KERNEL_SHA256` to `versions.env`. The generator reads from `versions.env`, so no further change is needed.
- A new tiny capture step is needed for the YARA Forge resolved tag: either at download time in `download-loki-yara-rules.sh` (write to `/var/lib/troskel/yara-forge-resolved-tag`), or at manifest-generation time (resolve again via HEAD; less reliable because the upstream may have rotated `latest` between download and manifest generation). At-download-time is the right answer.

### Propagation to the scanning host

`prepare-data-usb.sh` writes the manifest to the data USB alongside the scanner rootfs and signature files. `load-scanner` copies it from the data USB to `/var/lib/troskel/build-manifest.json` on the scanning host at boot, the same way it handles the other files in `/var/lib/troskel/`.

The manifest is then available to:

- `show-status` — a new line in the post-login status display showing the manifest's `generated_at` and Troskel commit SHA.
- The pre-login banner (planned in `definition-version-display.md`) — the Troskel commit SHA could be added there for at-a-glance identification of the build.
- The clean-certificate work (planned in `output-usb.md`) — the certificate references the manifest by commit SHA, giving the admin a chain from the certificate back to the exact build.

None of these consumers exist yet; the manifest is generated and propagated regardless, ready for those consumers to land.

## Side effects on existing scripts

- `prepare-build-machine.sh` gains a dependency on `clamav` (for `sigtool`). It is already conditionally installing the `clamav-freshclam` package; the parent `clamav` package brings `sigtool` along.
- `prepare-data-usb.sh` gains one line: copy the manifest from `/var/lib/troskel/` to the data USB.
- `load-scanner` gains one line: copy the manifest from the data USB to `/var/lib/troskel/` on the scanning host.
- `run-update.sh` gains a final step: invoke `generate-build-records.sh`.

No changes to the guest scripts, no changes to the verdict pipeline, no changes to the operator workflow.

## Estimated effort

One and a half days. The generator itself is the largest piece (around half a day, mostly in getting the JSON emission right — a heredoc with substituted variables, per the same dependency-minimisation principle that applies to the SBOM). The download-script tweaks are roughly an hour total. Propagation through `prepare-data-usb.sh` and `load-scanner` is half an hour. Documentation updates to `SECURITY.md` and `OPERATOR-GUIDE.md` are the rest.

## Sequencing

Depends on `checksum-verification.md` phase A being complete (the recorded SHA-256s the manifest emits are introduced there) and on phase B for the kernel hash. Phase A is in; phase B is in flight.

Closely coupled with `sbom-automation.md` — both are produced by the same generator. Land them in the same commit, with the shared generator and both output paths.

Target `1.0.0`. The principle the manifest serves — that the admin can reproduce a historical scan or explain a verdict change — is the operational property the project's auditability story rests on. Shipping `1.0.0` without it would mean shipping a security tool that cannot answer "what changed between yesterday's green and today's red". That gap is exactly the kind of thing the move to `1.0.0` should close.

## Open questions

- **Should the manifest include a hash of itself?** A self-hash makes the manifest tamper-evident in the trivial case (an attacker who changes a field but not the hash). It doesn't protect against an attacker who can re-emit the manifest. Probably not worth the complexity; GPG-signing the manifest (as part of the output-USB clean-certificate work) is the stronger property and is already planned.
- **Should the manifest be embedded in the scanner rootfs as well as the data USB?** This would let a forensic investigator extract the manifest from a running scanner image without access to the originating data USB. Useful but adds a dependency loop (the manifest documents the rootfs, but is now part of the rootfs). Defer; the data-USB copy is sufficient.
- **Should the YARA Forge resolved tag be written into `versions.env` like the kernel hash?** No: `versions.env` is "what was pinned by the developer"; the resolved tag is "what was downloaded by the build". Conflating them blurs the distinction. The manifest is the right place for resolved values.
- **What happens when an admin runs `prepare-data-usb.sh` against an existing data USB without re-running the full build?** The manifest on the data USB becomes stale relative to the data USB's actual contents. The simplest mitigation is for `prepare-data-usb.sh` to refuse if the manifest's `generated_at` is older than the youngest signature file by more than a small threshold. Practical, not perfect; revisit if this becomes an operational problem.
